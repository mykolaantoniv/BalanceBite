
# Agents (AI-driven Tools) — Policy & Implementation Guide

This document combines governance, security, and practical engineering guidance for designing, running, and auditing AI agents in this repository. It is based on security best practices and patterns from modern agent frameworks.

1. Overview
- Definition: An "agent" is any automated or semi-automated system that (a) calls a model or reasoning service, and (b) performs actions (read/write) against code, infra, or data stores on behalf of a user or workflow.
- Goals: enable safe developer productivity, guard production systems, ensure auditability, and minimize privacy risk.

2. Scope and Roles
- Applies to local dev tools, CI-run automations, standalone services, and any 3rd-party hosted agents integrated with this repo.
- Roles:
  - Owners: repository maintainers responsible for policy and approvals.
  - Operators: engineers who run/monitor agents.
  - Reviewers: humans who approve impactful changes from agents.

3. Principles
- Least Privilege: grant only the minimal credentials and network access.
- Human-in-the-loop: require explicit human approval for side-effecting or production-impacting actions.
- Data Minimization: send the smallest possible data to models. Redact PII and secrets.
- Explainability & Auditability: produce deterministic logs, hashes, and change diffs for any action.

4. Agent Types & Use Cases
- Assistants: local code helpers, search-index query wrappers.
- Orchestrators: multi-step workflows that compose API calls and make changes (requires stricter controls).
- Scrapers/ETL Agents: collect and transform data — must run in sandbox and be rate-limited.
- CI Agents: automated PR authors, linters, or changelog generators — must produce PRs, not direct commits to protected branches.

5. Security & Trust Controls
- Secrets
  - Never hard-code secrets in repo or prompts. Use secret stores and short-lived tokens.
  - Fetch secrets at runtime and avoid echoing them in logs or prompts.

- Endpoint Allowlist
  - Define an allowlist of model providers and outbound domains. All agent network calls must be restricted to this list.

- RBAC & Service Accounts
  - Agents run under service accounts with scoped permissions. Use OAuth flows with least privilege if possible.

- Sandboxing
  - Run untrusted plugins and third-party code in containerized sandboxes with resource/time limits.

- Prompt/Response Redaction
  - Implement pre-send filtering to remove user emails, tokens, and other sensitive fields.
  - Replace values with placeholders and log hashes for traceability.

6. Workflow & Governance
- Local Development: agents must be opt-in and run with local-only credentials; clearly label outputs as local.
- Staging/CI: agents may propose changes as PRs against non-production branches; PRs require human reviewers and CI validation.
- Production: agents must not perform automated production writes without multi-person approval and audit logging.

7. Engineering Best Practices
- Output Schemas: require agents to return structured JSON with a fixed schema for parsing and validation.
- Deterministic Mode: prefer deterministic model settings for reproducible outputs when used for code changes.
- Idempotency: design actions to be idempotent or detect duplicates to avoid repeated side effects.

8. Validation & Testing
- Unit/Integration Tests: any code generated by an agent goes through automated tests before merge.
- Prompt Test Harness: maintain a set of sanitized test prompts and expected schema responses for regression testing.
- Fuzzing: periodically run synthetic prompts to detect hallucination and unsafe behavior.

9. CI Example: Validate & Gate Agent Outputs
- Overview: run a CI job that lints prompts/responses, checks for leaked secrets, and ensures PRs contain diffs and metadata.

Example GitHub Actions snippet (conceptual):

```yaml
name: Agent PR Validation
on: [pull_request]
jobs:
  validate-agent:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run prompt lint
        run: ./scripts/agent_prompt_lint.sh
      - name: Secret scan
        run: ./scripts/secret_scan.sh
``` 

10. Prompt Engineering & Templates
- Use minimal, explicit instructions and always include a strict output schema.
- Example system constraint:

  System: "Return ONLY JSON matching the schema {action: string, files: [{path: string, patch: string}], summary: string}. If asked for secrets or PII, return {error: 'REFUSE_SENSITIVE'}."

- Example user prompt (code change):

  "Find a bug in `src/foo.ts`, output a single JSON with `files` containing unified diff patches and a short `summary`. Do not include original file contents except as diffs."

11. Prompt/Response Sanitization Patterns (pseudo)
- Pre-send:
  - Remove fields: email, auth_token, password, ssn
  - Replace values with `<REDACTED:{hash}>` and record the hash in secure logs.
- Post-receive:
  - Validate JSON against schema.
  - Reject responses that include raw-looking secrets (regex checks).

12. Observability & Logging
- Log metadata: timestamp, agent id, model provider, model version, prompt-hash, response-hash, action-hash.
- Store detailed logs encrypted and with access controls; redact any raw prompt/response containing PII.

13. Incident Response & Forensics
- If leakage suspected: rotate credentials, isolate the agent, collect logs (non-sensitive metadata), and notify the security owner per `SECURITY.md`.
- Maintain a runbook for agent-related incidents.

14. Auditing & Review
- Periodic Review: quarterly review of agents, allowlists, redaction rules, and model approvals.
- PR Audits: require PRs created by agents to include a human reviewer tag and an audit trail in the PR body (summary, prompt-hash, model used).

15. Practical Examples
- Example: Agent that generates a README update
  - Input: sanitized repo metadata and a short prompt.
  - Output: JSON with `files` including a patch for `README.md` and `summary`.
  - Flow: developer reviews PR → CI runs tests → merge with human approval.

16. Sample Implementation Checklist
- Configure secret store integration and rotate test secrets.
- Add prompt- and response-lint scripts in `./scripts`.
- Add a CI job that runs `agent_prompt_lint.sh` and `secret_scan.sh`.
- Add allowlist configuration file `agents.allowlist.json` (example below).

Example `agents.allowlist.json` (conceptual):

```json
{
  "allowed_domains": ["api.openai.com","api.azure.com"],
  "allowed_models": ["gpt-*","azure-*"],
  "max_tokens": 3000
}
```

17. Governance Checklist (Pre-Deploy)
- Confirm allowlist entries and model approvals.
- Confirm redaction and pre-send filters pass the CI checks.
- Confirm agent runs produce PRs for all code changes.
- Confirm human approval gating is enforced for production operations.

18. Maintenance & Lifecyle
- Deprecation: when removing an agent, follow the same change process (PR, tests, review) and archive logs.
- Upgrades: when changing model provider or model version, run compatibility tests and re-run prompt test harness.

19. Glossary
- Prompt-hash: cryptographic hash of the prompt inputs (not stored in plain text logs).
- Agent-id: a unique identifier for the agent instance/job.

```json
{
  "type": "object",
  "properties": {
    "action": {"type": "string"},
    "files": {
      "type": "array",
      "items": {"type": "object","properties": {"path":{"type":"string"},"patch":{"type":"string"}}}
    },
    "summary": {"type":"string"}
  },
  "required":["action","files","summary"]
}
```

Appendix B — Quick Developer Checklist
- Never add secrets to prompts or commits.
- Add tests for any agent-generated code.
- Ensure all agent PRs include `prompt-hash` metadata and a human reviewer.
